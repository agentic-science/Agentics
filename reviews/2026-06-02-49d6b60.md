## Findings

### P2: S3 promotion can leave durable orphaned objects after partial failure

Evidence:

- `crates/storage/src/s3.rs:173` implements S3 `promote` as `get_to_file`, `put_file`, then `delete(temporary_key)` at `crates/storage/src/s3.rs:187`.
- If temporary deletion fails after durable upload, `promote` returns an error while the durable object already exists.
- Solution submission cleanup deletes the DB row and temporary key only at `crates/services/src/solution_submissions.rs:115`.
- Private asset upload has the same pattern at `crates/services/src/challenge_drafts/private_assets.rs:160`.

Failure mode: a transient S3 delete failure after successful durable upload can leave a solution ZIP or private asset object under its durable key without a live DB row.
That is a storage-retention and quota leak.
Public exposure is residual, but bucket retention, backups, and operator access can still retain the object.

Recommendation: make `S3Storage::promote` return a clear invariant.
Either treat durable upload success as success and make temp deletion best-effort with `_tmp` cleanup, or return structured state that tells callers whether the durable object was created.
Do not blindly delete the durable key on all promote errors, because conflict errors may refer to an existing valid object.
Add a regression test that simulates delete failure after durable upload.

### P2: Submitter log retrieval prefers stale validation logs over official logs

Evidence:

- Admin official-run can queue official evaluation for an existing submission at `backend/api-server/src/handlers/admin.rs:175`.
- Submission fetches load both validation and official evaluations at `crates/persistence/src/db/solution_submissions.rs:621` and `crates/persistence/src/db/solution_submissions.rs:625`.
- Main evaluation projection is official-first at `crates/persistence/src/db/solution_submissions.rs:676`.
- The logs endpoint does the opposite: it returns validation logs immediately at `backend/api-server/src/handlers/artifacts.rs:92`, before applying official redaction logic at `backend/api-server/src/handlers/artifacts.rs:96`.

Failure mode: after a validation submission receives an official run, `/logs` can return old validation diagnostics instead of eligible official diagnostics, or report `available` instead of the correct official redaction state for a private official run.

Recommendation: add an explicit `eval_type=validation|official` selector, or make the default match the official-first result projection.
Add tests for a validation submission later official-run on both public-only and private official contracts.

### P2: Identity and auth workflows still live in HTTP handlers

Evidence:

- `register_agent` generates tokens, handles pioneer-code parsing, and calls persistence directly in `backend/api-server/src/handlers.rs:87`.
- Admin login, GitHub OAuth state/session creation, creator upsert, cookies, and GitHub HTTP calls are in `backend/api-server/src/auth_handlers.rs:165`, `backend/api-server/src/auth_handlers.rs:235`, and `backend/api-server/src/auth_handlers.rs:417`.
- The architecture docs say API handlers should be thin and services should own use cases in `docs/architecture/en.md:121` and `docs/architecture/en.md:141`.

Failure mode: auth policy, provider IO, persistence, cookie/session semantics, and error mapping are hard to test or reuse independently.
This is confirmed coupling rather than a confirmed runtime bug.

Recommendation: move registration, admin session, and GitHub OAuth workflows into `agentics-services::auth` or `agentics-services::identity`.
Inject a GitHub OAuth client so callback behavior can be tested without live provider IO.

### P3: Bounded setup/build workspace copies silently drop non-regular files

Evidence:

- `copy_dir_all_blocking` recurses through directories and copies regular files only at `crates/runner/src/filesystem.rs:268`; it has no error branch for symlinks, sockets, FIFOs, or devices.
- Bounded setup/build copies retained workspaces before each phase at `crates/runner/src/topologies.rs:125`.
- Evaluator-visible run tree copy already rejects symlinks and non-regular files at `crates/runner/src/run_io.rs:159` and `crates/runner/src/run_io.rs:180`.

Failure mode: setup or build can create unsupported filesystem entries that are silently omitted in bounded slot mode.
The next phase receives a different tree than the previous phase produced, creating mode-dependent behavior and hiding invalid output.

Recommendation: make setup/build workspace copy strict with `symlink_metadata`; reject symlinks and non-regular entries with phase-scoped runner errors.
Add bounded-mode tests for symlink and special-file output.

### P3: Ranking context and score distribution silently truncate at 10,000 rows

Evidence:

- Ranking context reads only 10,000 leaderboard rows at `crates/services/src/public_projection/leaderboard.rs:94`.
- It sets `total_ranked` from that limited vector at `crates/services/src/public_projection/leaderboard.rs:101`.
- Score distribution has the same cap at `crates/services/src/public_projection/score_distribution.rs:26` and derives count from truncated values at `crates/services/src/public_projection/score_distribution.rs:54`.

Failure mode: if a challenge exceeds 10,000 ranked agents, rank, total ranked, percentiles, histogram count, and quantiles become exact-looking but wrong.
Default active-agent config is lower, so this is a scale risk rather than an MVP blocker.

Recommendation: compute rank and counts with SQL window functions, fetch only nearby rows for ranking context, and either compute score distributions over the full visible set or expose explicit sampling metadata.
