## Findings

### P1: Matrix official benchmark data is reconstructable from public code

Refs:

- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/v1/evaluator/prepare.py:19`
- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/v1/evaluator/prepare.py:97`
- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/tools/generate_assets.py:92`
- `backend/api-server/src/challenge_creation_handlers.rs:955`

Problem: the matrix evaluator has public fallback official seeds and silently uses them when `private-benchmark/config.json` is absent. The private asset generator also uses fixed public seeds. An uploaded required asset with the right asset name but missing the expected config can still publish, then official prepare falls back to public default dimensions/seeds.

Impact: agents can reconstruct official inputs, defeating the private benchmark assumption.

Fix: require `private-benchmark/config.json` in official mode, remove public official defaults, and generate private seeds from non-committed input. Add a regression where a declared required private seed asset without the expected config fails validation or publish.

### P1: Hosted DGX policy rejects the current bundled challenge image refs

- `deploy/dgx-spark/agentics.env.example:29`

### P2: Draft validation does not validate the assembled runtime bundle

Refs:

- `backend/api-server/src/challenge_creation_handlers.rs:775`
- `backend/shared/src/challenge_creation.rs:242`
- `backend/api-server/src/challenge_creation_handlers.rs:709`
- `backend/api-server/src/challenge_creation_handlers.rs:717`

Problem: draft validation validates the public repository and computes a review bundle hash, but only publish assembles private assets into a runtime bundle and runs `challenge_bundle::validate_challenge_bundle`.

Impact: a draft can pass validation and approval, then fail during publish because private overlay evaluator or prepare paths are broken.

Fix: draft validation should assemble a temporary runtime bundle with active private assets, validate it with `challenge_bundle::validate_challenge_bundle`, run digest-pinned image checks when required, and store the reviewed runtime bundle hash from that assembled content.

### P2: Permission-repair containers can inherit image ENTRYPOINT

Refs:

- `backend/shared/src/runner/docker.rs:75`
- `backend/shared/src/runner/docker.rs:214`
- `backend/shared/src/runner/docker.rs:541`

Problem: permission repair reuses the phase image and sets only `cmd`, not an explicit entrypoint, while exposing writable bind mounts and `FOWNER`.

Impact: an image ENTRYPOINT can intercept or bypass the intended chmod command, mutate writable mounts after the main phase exits, or leave root-owned files that break cleanup.

Fix: explicitly clear or set the entrypoint for phase and repair containers, and consider a platform-owned minimal repair image or host-side repair routine.

### P2: Evaluator receives the whole writable `/io` tree

Refs:

- `backend/shared/src/runner.rs:724`
- `backend/shared/src/runner.rs:754`
- `backend/shared/src/runner.rs:792`
- `backend/shared/src/runner.rs:796`
- `backend/shared/src/runner.rs:808`
- `backend/shared/src/runner/run_io.rs:154`

Problem: participant run containers get writable `/io` for home/tmp/output. After a run, the runner verifies declared outputs, but the evaluator-visible tree is the whole `/io` tree.

Impact: solution code can pass arbitrary extra regular files to evaluator code within output limits. That expands the declared output contract and increases attacker-controlled evaluator surface.

Decision: accepted for MVP. Document clearly that the evaluator receives a hostile, participant-controlled run tree, and that challenge-owned evaluator code must ignore unexpected files and read only platform metadata plus declared outputs.

### P2: Public result DTOs do not carry the metric name for `official_score`

Refs:

- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/v1/spec.json:115`
- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/v1/evaluator/run.py:83`
- `backend/shared/src/db/evaluations.rs:211`
- `frontends/web/src/lib/metrics.ts:31`
- `frontends/web/src/app/(observer)/challenges/[name]/leaderboard/page.tsx:183`

Problem: matrix declares `total_wall_time_ms` as the primary ranking metric, but evaluator `primary_score` is correctness. The frontend wraps `official_score` as if it were the declared primary metric because the public DTO does not tell it which metric name the score represents.

Impact: a correctness value can render as milliseconds under “Total Wall Time,” or any future non-aggregate score can be mislabeled as the ranking metric.

Fix: backend public result/leaderboard/submission DTOs should carry explicit primary-score metadata, including the score metric name and display label. The frontend should render `<metric-name> (Primary)` from backend-provided metadata instead of inferring the metric name from `metric_schema.ranking.primary_metric_name`.

### P2: Private overlay generator leaves validator-forbidden residue

Refs:

- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/tools/generate_assets.py:92`
- `challenge-repos/agentics-challenges/challenges/matrix-multiplication/.gitignore:1`
- `challenge-repos/agentics-challenges/scripts/validate_challenges.py:216`
- `backend/integration-tests/tests/matrix_multiplication.rs:35`

Problem: following the sample tool to create the private ZIP leaves `v1/private-benchmark/` under the challenge root. That directory is the private overlay source for `datasets.private_benchmark_dir`; in this challenge it contains `private-benchmark/config.json`, which official prepare uses to generate hidden official inputs and expected outputs. It is useful as ZIP content, but it must not remain in the public proposal checkout. The repository validator rejects that path, and the integration test encodes a hidden cleanup.

Impact: creators can follow the documented/sample flow and end up with a locally invalid proposal unless they manually remove generated private residue.

Decision: add repository `.gitignore` guardrails for `private-benchmark/`, `private_benchmark/`, and generated private overlay ZIPs, but keep backend draft validation strict because force-added private files can still exist in a Git checkout. Update the local challenge repository validator to respect `.gitignore` or scan tracked files only, so ignored locally generated private overlays do not block local validation. The generator may still stage private config outside the challenge root or clean it up after writing the ZIP, but ignored local residue should be treated as a developer convenience rather than publishable material.
