## Findings

### P1: Admin service tokens survive admin-role revocation

Status: Open.

Evidence:

- Creating an admin service token requires a current human admin in `backend/api-server/src/handlers/admin.rs:119`.
- `revoke_admin_role` only updates `human_roles` in `crates/persistence/src/db/sessions.rs:478` and does not revoke active `admin_service_tokens` created by that human.
- Admin service-token authentication checks only token status and expiry in `crates/persistence/src/db/sessions.rs:626`.
- `AdminAuth` accepts bearer-token admin actors in `backend/api-server/src/extractors.rs:172`, and admin mutations such as rejudge and agent disable accept `AdminAuth` at `backend/api-server/src/handlers/admin.rs:220` and `backend/api-server/src/handlers/admin.rs:286`.

Failure mode:

A human admin can mint a service token, lose the admin role, and keep performing operational admin actions through the still-active token.

Remediation:

Either authenticate service tokens by joining the creator human and active admin role, or revoke owner-created admin service tokens transactionally on admin-role revocation and account disablement.
If durable non-human service accounts are intended, model them separately from human-owned admin privilege.

### P1: Pioneer codes are low-entropy plaintext secrets

Status: Open.

Evidence:

- `PioneerCode::generate` uses four random bytes in `crates/domain/src/models/pioneer_codes.rs:144`.
- Validation accepts an 8-hex random suffix in `crates/domain/src/models/pioneer_codes.rs:325`.
- Code creation stores both the hash and plaintext display value in `crates/services/src/admin.rs:93`.
- The initial migration persists `code_display TEXT NOT NULL UNIQUE` in `backend/migrations/001_identity_and_platform.sql:96`.
- Admin list/detail presentation returns `code_display` in `crates/services/src/admin.rs:434`.

Failure mode:

Registration/setup gate secrets have about 32 bits of entropy and remain recoverable from the database and admin API responses.
The failed-code delay reduces online guessing speed, but it is not a substitute for strong secret entropy or one-time secret display.

Remediation:

Generate at least 128-bit URL-safe random codes, store only hashes, return the raw code only in the creation response, and use a non-secret display surrogate such as prefix plus last four characters for list/detail views.

### P2: Agent registration response can expose bearer tokens through `Debug`

Status: Open.

Evidence:

- `RegisterAgentResponse` derives `Debug` while carrying raw `token: String` in `crates/domain/src/models/request.rs:77`.
- The registration service places the generated bearer token into that DTO in `crates/services/src/auth.rs:299`.
- Admin and creator token creation responses already use custom redacted `Debug` implementations in `crates/domain/src/models/auth.rs:269` and `crates/domain/src/models/auth.rs:325`.

Failure mode:

No current log site was found that prints this DTO, so this is residual risk.
The type itself still violates the project rule that secrets must not appear in debug output.

Remediation:

Implement a redacted `Debug` for `RegisterAgentResponse` and consider carrying the token in a secret wrapper until the JSON response boundary.

### P2: Evaluation workers keep running after failing to persist the start claim

Status: Open.

Evidence:

- `run_one_cycle` logs when `mark_started` returns false in `crates/services/src/evaluation_lifecycle.rs:132`, but still starts the lease task and proceeds to runner execution at `crates/services/src/evaluation_lifecycle.rs:153`.
- `mark_evaluation_started` returns false when the job no longer matches worker, attempt, target, eval type, or running status, or when the evaluation row already exists, in `crates/persistence/src/db/evaluations.rs:36`.
- Final writes are guarded by worker and attempt in `crates/persistence/src/db/evaluations.rs:107`, so this does not look like a result-corruption path.

Failure mode:

A stale or lost-claim worker can still consume runner/container capacity even though its result will be discarded.

Remediation:

Treat `mark_started == false` as a lost claim and skip execution, or re-read the job to distinguish an idempotent existing start row from stale ownership.
Add a regression test where the claim is requeued before `mark_started`.

### P2: Runner containers are writable-rootfs and do not enforce a non-root user

Status: Open.

Evidence:

- Runner container creation passes `false` for `readonly_rootfs` in `crates/runner/src/docker.rs:710`.
- It relies on Docker layer quota via `storage_opt` in `crates/runner/src/docker.rs:719`.
- The created container config does not set a `user` in `crates/runner/src/docker.rs:723`.
- A chmod repair path exists for files created by root-running images in `crates/runner/src/docker.rs:253`.

Failure mode:

Hostile solution or evaluator code can write into the container layer and runs as the image default user when images default to root.
Production mitigates this with Docker layer quota, host probes, capability drops, no-new-privileges, pid caps, and resource limits, but writable rootfs plus image-default root leaves more blast radius if quota enforcement is unavailable or regresses.

Remediation:

Run normal participant/evaluator phases with `readonly_rootfs: true`, provide explicit writable tmpfs or bind mounts for `/tmp`, `HOME`, and caches, set a fixed non-root UID/GID in container config or base images, and keep a startup probe that proves layer-quota enforcement by attempting an over-limit write.

### P2: Bundle tar unpack lacks independent entry and copied-byte caps

Status: Open.

Evidence:

- `unpack_tar_to_directory_blocking` validates tar paths and entry type, then creates files without counting entries or copied bytes in `crates/storage/src/tar_archive.rs:197`.
- The runner downloads a size-capped tar and immediately unpacks it in `crates/runner/src/execution.rs:93`.
- Normal publishing caps pack output before archive creation in `crates/storage/src/tar_archive.rs:73`.

Failure mode:

The normal pack path is capped, but a tampered or corrupted stored bundle can still force high file-count or sparse-file stress during runner extraction within the stored-object byte cap.

Remediation:

Make `unpack_tar_to_directory` accept an archive policy, count entries and copied bytes, reject sparse/hardlink/link/device entries explicitly, enforce max depth/files/expanded bytes while copying, and add hostile-tar regression tests.

### P2: Leaderboard repair/upsert erases typed identifiers inside persistence

Status: Open.

Evidence:

- `LeaderboardRepairScope` stores `challenge_name`, `target`, and `agent_id` as raw `String` in `crates/persistence/src/db/leaderboard.rs:40`.
- The repair scope reads `(String, String, String)` directly from DB rows in `crates/persistence/src/db/leaderboard.rs:51`.
- `LeaderboardReplacementCandidate` stores `id: String` in `crates/persistence/src/db/leaderboard.rs:302`.
- Upsert reads raw `challenge_name` and `agent_id` and passes them into `lock_leaderboard_scope` in `crates/persistence/src/db/leaderboard.rs:424` and `crates/persistence/src/db/leaderboard.rs:531`.

Failure mode:

Leaderboard repair and write paths can operate on unparsed challenge names, target names, agent IDs, and submission IDs after the DB boundary.
Current DB constraints reduce exploitability, but corrupted rows or future call sites would produce cast/runtime failures or stale/wrong repair behavior instead of precise domain errors.

Remediation:

Make `LeaderboardRepairScope` and replacement candidates use `ChallengeName`, `TargetName`, `AgentId`, and `SolutionSubmissionId`; parse rows through existing row helpers; and accept typed values in lock/upsert helpers.

### P2: Challenge-review repositories downcast parsed record IDs back to `&str`

Status: Open.

Evidence:

- Repository methods take raw IDs in `crates/persistence/src/repositories/challenge_review_records.rs:32`, `crates/persistence/src/repositories/challenge_review_records.rs:40`, `crates/persistence/src/repositories/challenge_review_records.rs:208`, `crates/persistence/src/repositories/challenge_review_records.rs:221`, and `crates/persistence/src/repositories/challenge_review_records.rs:247`.

- DB functions continue the raw boundary at `crates/persistence/src/db/challenge_creation.rs:169` and `crates/persistence/src/db/challenge_creation/publishing.rs:68`.

Failure mode:

Callers past the HTTP extractor can bypass the typed ID contract and rely on SQL `::uuid` casts for validation.

### P2: Production Compose port envs drift from service listen ports

Status: Open.

Evidence:

- Production Compose publishes API traffic to container port `3100` in `deploy/compose/compose.prod.yml:136`, while the API binds `config.api_web.api_port` in `backend/api-server/src/main.rs:74`.
- Production Compose publishes web traffic to container port `3001` in `deploy/compose/compose.prod.yml:192`, and the web command hard-codes `-p 3001` in `deploy/compose/compose.prod.yml:193`.
- The web env loader requires and parses `AGENTICS_WEB_PORT`, then discards it in `frontends/web/src/lib/env.ts:28`.
- Docs describe `AGENTICS_API_PORT` and `AGENTICS_WEB_PORT` as listen ports in `docs/ports-and-paths/en.md:13`.

Failure mode:

Changing `AGENTICS_API_PORT` in production makes the API listen on a different port than Compose forwards.
Changing `AGENTICS_WEB_PORT` does not change the web process listen port, and may instead affect only host-publish fallback semantics through Compose interpolation.

Remediation:

Split service listen ports from host publish ports, or make Compose targets and commands follow the same envs.
If `AGENTICS_API_HOST_PORT` and `AGENTICS_WEB_HOST_PORT` are the intended production knobs, document them as host-publish controls and keep service ports fixed.

### P3: Auth-dependent UI can render stale signed-in state after session revalidation failure

Status: Open.

Evidence:

- `useCreatorSession` returns `swr.data` without clearing or masking it on `getHumanSession` error in `frontends/web/src/lib/creatorData.ts:13`.
- `useAdminSession` does the same in `frontends/web/src/lib/adminData.ts:53`.
- `AccountMenu`, `SignInPanel`, and `AccountSettingsPanel` render auth-dependent UI from the same SWR data in `frontends/web/src/components/AccountMenu.tsx:27`, `frontends/web/src/components/SignInPanel.tsx:21`, and `frontends/web/src/components/AccountSettingsPanel.tsx:25`.

Failure mode:

If a session expires, is revoked, or is deleted outside the current tab, SWR can retain stale data after failed revalidation and keep signed-in/account/admin controls visible until explicit cache mutation or reload.
Backend CSRF/session checks still gate mutations, so this is primarily privacy and UX risk.

Remediation:

Centralize a session hook that treats 401/403 as signed-out, mutates `HUMAN_SESSION_CACHE_KEY` to `undefined`, and add a focused test that seeds stale session data then forces `getHumanSession` to reject.
