# Full Code Review - 2026-05-21 - `adb953e`

Review scope: current `main` after `coexecuted_benchmark` implementation and the `agentics-challenges` validator update.

## Decisions Recorded

- Private benchmark data plus network-enabled official participant-containing stages is an expected trust-boundary risk for now.
  Do not hard-reject this configuration yet. Add explicit warnings in challenge checking scripts when these stages enable network access.
- Validation must not run from a private runtime bundle.
  Reject validation when `private_benchmark_enabled` is true and `public_bundle_path == bundle_path`.
- Submission hiding is admin moderation and is out of MVP scope.
  Remove the hide-submission route, UI action, DTO, tests, and docs references instead of adding a new moderation state.
  Keep `visible_after_eval` as evaluation lifecycle visibility only.
- Agent disable remains in MVP as an operational emergency control, not as a broader moderation system.
- The `agentics-challenges` Python validator should temporarily port the exact Agentics target, image, execution-mode, and executor-field policy.
- Longer term, the challenge repository should reuse Agentics Rust validation and remove the duplicated Python validator.
  Tracked as https://github.com/agentic-science/agentics-challenges/issues/2.

## Findings

### P1 - Official private benchmark data can be exposed by network-enabled participant-containing stages

Status: accepted risk with required warnings.

Evidence: official runs select the private bundle in `backend/shared/src/runner.rs`, separated official inputs are mounted into participant run containers, and `coexecuted_benchmark` mounts `/challenge` and `/workspace` into the same evaluator-image container.
Stage `network_access` flows from the challenge-owned resource profile into Docker network mode.

Failure mode: if a challenge owner enables outbound network for a stage that contains participant code and private benchmark material, a malicious solution can exfiltrate private inputs or benchmark files.

Decision: do not reject this for MVP. Add warnings in checking scripts when the risky combination is present.

### P1 - Validation can receive private bundles for seeded or backfilled challenges

Status: confirmed.

Evidence: validation uses `payload.public_bundle_path`, but migration/backfill and seeded challenge sync can set `public_bundle_path = bundle_path`.

Failure mode: validation jobs can mount a full private runtime bundle as `/challenge`, especially dangerous for `coexecuted_benchmark` because participant-built workspace and the coexecuted-evaluator share a container.

Remediation: reject validation admission when `private_benchmark_enabled` is true and `public_bundle_path == bundle_path`.
Add a regression test proving validation cannot see private benchmark paths in that state.

### P1 - Submission hiding conflicts with evaluation lifecycle visibility

Status: confirmed; remove from MVP.

Evidence: admin hide updates `visible_after_eval = false`, while official completion and stale-job maintenance can later set `visible_after_eval = true`.

Failure mode: if an admin hides a submission while an official rerun is active, or if stale-job repair runs later, the submission can become public again.

Decision: do not add admin moderation state for MVP. Remove the hide-submission admin route and UI action.
Keep `visible_after_eval` as evaluation-derived visibility only. Keep agent disable as an operational emergency control.

### P1 - Challenge repository validator rejects valid execution specs

Status: confirmed.

Evidence: `challenge-repos/agentics-challenges/scripts/validate_challenges.py` uses a generic validation-source check that rejects direct `coexecuted_benchmark` validation and does not accept `piped_stdio` `validation_session`.

Failure mode: valid challenge specs can fail CI even though the backend and worker accept them.

Remediation: make the Python validation-source matrix mode-specific: `separated_evaluator` accepts `validation_runs` or `validation_setup`, `piped_stdio` accepts `validation_session` or `validation_setup`, and `coexecuted_benchmark` accepts direct benchmark validation with optional `validation_setup`.

### P1 Risk - Hosted worker safety depends on local-looking API bind settings

Status: risk.

Evidence: default writable storage is unbounded and Docker layer quota is off.
Runner storage validation permits unbounded storage when `api_host` is loopback.

Failure mode: a production-connected worker launched with default local-looking bind settings can process untrusted submissions with unbounded bind mounts and writable container layers.

Remediation: make worker runner safety independent of API bind host.
Require an explicit local or hosted runner profile and fail closed for hosted workers unless quota slots, Docker layer quota, and host probes are enabled.

### P2 - Admin rejudges count against participant quotas

Status: confirmed.

Evidence: submission quotas count evaluation jobs, and admin rejudge inserts additional jobs for an existing submission.

Failure mode: administrative retries can consume participant submission quota even though the participant did not submit a new artifact.

Remediation: model quota consumption explicitly.
Count only participant-created quota-charged submissions or jobs, while keeping active-capacity accounting separate.

### P2 - Generated Zod schemas weaken `RunName` to `any`

Status: confirmed.

Evidence: Rust constrains `RunName`, but generated frontend schemas emit `run_name: z.any().refine(...)`.

Failure mode: frontend API validation accepts non-string run names, which can reach React rendering paths.

Remediation: fix schema generation or add a `RunName` override that starts from `z.string()`, then add negative schema tests for non-string and unsafe names.

### P2 - Challenge repository validator is weaker than the Rust contract

Status: confirmed.

Evidence: the Python validator does not enforce supported image repositories, CPU/CUDA tag policy, CUDA metadata, digest policy, or trusted executor `result_file`.

Failure mode: challenge PR CI can pass specs that publish-time Rust validation later rejects.

Decision: temporarily port the exact Rust policy into the Python validator.
Long-term replacement is tracked in https://github.com/agentic-science/agentics-challenges/issues/2.

### P2 - Review docs and skills have stale coexecuted and digest guidance

Status: confirmed.

Evidence: reviewer guidance still implies prepare result manifest paths for all modes, but `coexecuted_benchmark` prepare forbids result locators.
Review docs also describe the older approval digest model instead of the assembled private runtime bundle plus separate public-only bundle.

Failure mode: reviewers can reject valid coexecuted drafts or misunderstand what approval freezes.

Remediation: update English and Chinese review docs plus author/reviewer skills with mode-specific prepare rules and current public/private bundle behavior.

### P2/P3 - Client DTO boundaries do not enforce coexecuted `solution.run` invariant

Status: risk.

Evidence: generated web schemas make `solution.run` generally optional, independent of `execution.mode`, and CLI deserializes public DTOs without a post-parse topology invariant.

Failure mode: stale DB rows or fixture drift can make clients accept and render an impossible coexecuted topology with a participant run container.

Remediation: add post-parse invariant checks in the web schema facade and CLI DTO handling, or expose a shared Rust validation helper for public challenge DTOs.

### P3 - Creator API helpers rely on callers for request validation

Status: confirmed.

Evidence: some web creator API helpers stringify caller-provided requests directly while validation is performed in UI components.

Failure mode: future callers can bypass runtime request validation.

Remediation: validate request bodies inside exported API helpers before `JSON.stringify`, keeping UI validation as early feedback only.

### P3 - Coexecuted rendering does not surface the trust boundary

Status: confirmed.

Evidence: web and CLI challenge-detail rendering show the raw mode and command metadata but do not highlight that `coexecuted_benchmark` has no separate participant run container and has a weaker official private-benchmark boundary.

Failure mode: users see raw topology data without the operational warning they need to interpret the challenge contract.

Remediation: add localized mode labels and a coexecuted warning or metadata row in web and CLI rendering.

### P3 - Coexecuted schema and rendering tests are positive-only

Status: confirmed.

Evidence: current contract fixtures are mostly separated-evaluator based, while coexecuted coverage is hand-built positive parsing.

Failure mode: tests can pass while missing Rust-serialized coexecuted DTOs, negative `solution.run` invariants, and user-facing rendering coverage.

Remediation: add Rust-generated coexecuted DTO fixtures, web facade negative tests, and CLI/web rendering tests for the benchmark topology.

## Areas Reviewed Without New Findings

- Worker stale-claim result writes are guarded by current claim identity.
- Official logs and public result redaction remain centralized enough for the reviewed paths.
- Archive extraction, storage key validation, and artifact summary handling still reject traversal, symlinks, duplicate entries, and unsafe storage paths.
- Piped stdio transcript and stderr caps remain bounded.
- Submission and job creation admission controls still use transactional capacity checks.
