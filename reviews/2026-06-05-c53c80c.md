## Findings

### P1: Creator tokens survive pioneer-code revocation

Status: Fixed.

Evidence:

- `crates/persistence/src/db/pioneer_codes.rs` revoked human sessions, creator roles, admin service tokens, agents, and agent tokens, but did not revoke `creator_api_tokens`.
- `crates/persistence/src/db/sessions.rs` authenticates creator API tokens from their token row and current human status/roles.
- `backend/api-server/src/extractors.rs` accepts active creator/admin humans behind those tokens.

Failure mode:

If a human receives Creator through a pioneer code, creates a creator API token, and that pioneer code is later revoked, the token becomes unusable while the human is `setup_required`, but can become valid again if the same human later regains Creator/Admin access. That violates the revocation lifecycle.

Remediation:

Revoke active `creator_api_tokens` for affected humans in the same pioneer-code revoke transaction. Expose `revoked_creator_api_token_count` in the revoke response and cover revoke, re-setup, and old-token rejection in integration tests.

### P1: CLI config persistence can put creator tokens in argv

Status: Fixed.

Evidence:

- `frontends/agentics-cli/src/cli.rs` accepted `agentics config set creator-api-token <value>`.
- `frontends/agentics-cli/src/commands.rs` stored that positional value.
- Docs recommended `agentics config set creator-api-token` while also saying creator tokens must stay out of argv/logs.

Failure mode:

Long-lived creator API tokens can leak through shell history, process listings, CI logs, terminal capture, or command audit tools.

Remediation:

Reject positional values for secret config keys and require `--stdin` for `token` and `creator-api-token`. Update CLI errors, English/Chinese docs, and the challenge-authoring skill.

### P1: Admin review validation output hides the digest needed for approval

Status: Fixed.

Evidence:

- `frontends/agentics-cli/src/output/review_records.rs` table output showed status and counts but not `validation_bundle_sha256`, `approved_bundle_sha256`, `validation_message`, or latest validation record details.
- Reviewer docs instruct admins to use the validation digest as `expected_validation_bundle_sha256`.

Failure mode:

`agentics admin review-record validate ...` can report `status: validated` without giving the operator the digest needed for the next documented approval command unless they know to rerun with `--json`.

Remediation:

Render validation/approval digests, validation messages, and latest validation record status/message in default table output. Add CLI coverage asserting the digest appears in `admin review-record validate` output.

### P1: One-time creator token remains visible after copy or access changes

Status: Fixed.

Evidence:

- `frontends/web/src/components/creator/CreatorConsole.tsx` kept `createdToken` in React state and DOM after copy.
- Rendering was gated only by `createdToken`, not by the currently signed-in creator identity/access.

Failure mode:

The raw one-time creator token can remain visible/copyable after copy, sign-out, account switch, or access downgrade.

Remediation:

Clear the raw token after successful copy, provide a dismiss action, clear it when creator identity/access changes, and keep it visible only when clipboard copy fails so the user can manually copy it.

### P1: Creator token SWR cache is not scoped to the signed-in human

Status: Fixed.

Evidence:

- `frontends/web/src/lib/creatorData.ts` used the constant key `"creator-api-tokens"`.

Failure mode:

Another creator in the same browser session can briefly see the previous creator's token labels, ids, statuses, and timestamps before revalidation.

Remediation:

Scope the SWR key by non-secret `human_id` and clear creator-token caches on sign-out.

### P2: CLI exposes admin lifecycle commands under the creator namespace

Status: Fixed.

Evidence:

- `frontends/agentics-cli/src/cli.rs` exposed validation, approval, rejection, publish, abandon, and cleanup under `challenge-creator review-record`.
- The implementation authenticated those variants with admin service tokens.
- Docs present these operations under `agentics admin review-record`.

Failure mode:

The command surface blurs creator and admin token classes and makes review lifecycle operations look creator-owned.

Remediation:

Remove admin lifecycle variants from `ChallengeReviewRecordCommand`. Keep them only under `AdminReviewRecordCommand` and move tests to `admin review-record`.

### P2: CLI sends bearer tokens to remote HTTP origins

Status: Fixed.

Evidence:

- `frontends/agentics-cli/src/config.rs` accepted any `http://` API base URL.
- CLI creator/admin request helpers attach bearer tokens to those requests.

Failure mode:

A misconfigured remote `http://agentics.example` base URL can leak creator/admin bearer tokens in cleartext.

Remediation:

Allow `http://` only for localhost/loopback by default. Require explicit `AGENTICS_ALLOW_INSECURE_REMOTE_HTTP=true` for non-loopback HTTP, and document that it is only for disposable test environments.

### P2: Creator token request validation returned generic bad request for invalid expiry

Status: Fixed.

Evidence:

- `crates/services/src/creator.rs` parsed creator token `expires_at` with `ServiceError::BadRequest`.

Failure mode:

Malformed or already expired token expiry values do not use the request-validation envelope/status expected for semantic request-body errors.

Remediation:

Return `validation_failed` for malformed expiry values and reject past expiry times.

### P2: Raw token creation responses are debug-printable

Status: Fixed.

Evidence:

- `CreatorApiTokenCreatedResponse` and `AdminServiceTokenCreatedResponse` derived `Debug` while carrying raw one-time tokens.

Failure mode:

Future debug logs, panic output, or test snapshots could expose a one-time creator/admin token.

Remediation:

Implement redacted `Debug` for both token creation responses and add a domain regression test.

### P2: Deprecated creator web workflow panels remain committed

Status: Fixed.

Evidence:

- `frontends/web/src/components/creator/CreatorForms.tsx` and `CreatorPanels.tsx` still contained PR registration, private asset, owner stats, participants, and shortlist UI after the creator console was reduced to identity/token management.
- `frontends/web/messages/*` still contained the matching creator workflow strings.

Failure mode:

Stale browser workflow components can be accidentally re-imported and keep the old web-first contract alive.

Remediation:

Extract only `CreatorIdentityPanel`, delete old creator workflow panels/tests, and trim stale creator workflow translations.

### P2: Creator token table renders raw statuses and timestamps

Status: Fixed.

Evidence:

- `frontends/web/src/components/creator/CreatorConsole.tsx` rendered `active` and raw ISO timestamps directly.

Failure mode:

Language-switched UI still shows raw internal values.

Remediation:

Render localized status labels and locale-formatted dates.

### P2: CLI workflow coverage remains incomplete

Status: Partially addressed; follow-up recommended.

Evidence:

- Existing CLI tests did not cover every creator/admin operational command, output mode, and token-source precedence path.

Failure mode:

CLI-first workflows can drift from documented routes or output contracts.

Remediation in this pass:

Added focused coverage for the bugs fixed here: secret config positional rejection, admin review validation digest output, admin namespace validation, and non-UTF-8 repository path rejection under the admin namespace.

Recommended follow-up:

Split the already large CLI test module before adding exhaustive workflow coverage, then add table/JSON tests for creator stats, participants, shortlist show/upload, admin approve/reject/publish/abandon/cleanup/private-assets, and stdin/env/config token precedence.
