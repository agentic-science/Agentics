## P1 Findings

### P1: Storage no-overwrite semantics are not atomic

Confirmed.

References:

- `crates/storage/src/lib.rs:122`
- `crates/storage/src/lib.rs:566`
- `crates/storage/src/lib.rs:631`
- `crates/storage/src/lib.rs:887`
- `crates/storage/src/lib.rs:902`

Failure mode:

The `Storage` trait promises that `put`, `put_file`, and `promote` do not overwrite existing durable objects. Local storage checks existence, writes a temporary file, checks again, then renames into place. Unix rename can replace the destination if another writer creates it after the final check. S3 `put` only uses `If-None-Match: *` when no custom endpoint is configured, so RustFS/S3-compatible endpoints use check-then-put semantics. S3 `promote` checks destination existence and then uses `copy_object`, which can overwrite the destination if another writer wins the race.

Impact:

This can replace durable challenge bundles, public bundles, private assets, solution artifacts, runner logs, or shortlist JSON after DB admission has bound the object key. That breaks the storage layer's immutable-key assumption and can corrupt challenge/evaluation state under retries or concurrent requests.

Recommended fix:

Make the no-overwrite contract strict for every backend. Do not keep a "best effort unless S3-compatible" mode.

For local storage, write to a temp file and finalize with atomic no-replace semantics. A practical portable approach for the current Linux/macOS targets is `hard_link(temp, final)` followed by temp removal; hard-link creation fails if `final` already exists, so concurrent writers cannot replace the object.

For S3/RustFS, use conditional destination creation for every durable write, including custom endpoints. `put` and `put_file` should always use `If-None-Match: *`. `promote` should not use check-then-`CopyObject`; for MVP, stream the temporary object into a destination `PutObject` with `If-None-Match: *`, verify the destination length with `HeadObject`, then delete the temporary object. This is less efficient than server-side copy, but it is correct and portable. If a configured S3-compatible backend does not honor the condition, fail the storage test and treat that backend as unsupported until fixed. AWS-specific server-side promotion can be optimized later only if destination-absent conditional copy semantics are proven.

Concrete test coverage:

- Concurrent local `put`/`put_file` to the same key: one succeeds and one returns conflict.
- Concurrent local `promote` to the same durable key: one succeeds and one returns conflict.
- RustFS/S3 concurrent `put` and `promote` to the same durable key: one succeeds and one returns conflict.

### P1: Successful challenge publish leaves private runtime bundles in scratch storage

Confirmed.

References:

- `crates/services/src/challenge_drafts.rs:724`
- `crates/services/src/challenge_drafts.rs:1010`
- `crates/services/src/challenge_drafts.rs:1075`

Failure mode:

`publish_challenge_draft` removes `temporary_bundle_path` and `temporary_public_bundle_path` only when publish fails. On successful `new_challenge` publish, the assembled runtime bundle remains under `_tmp/challenge-bundles`; this bundle includes private asset overlays.

Impact:

Private benchmark data can remain on the host outside the durable object lifecycle. On hosted systems, this creates unnecessary private-data residue and storage pressure.

Concrete test coverage:

- Preserve the existing failure cleanup behavior.

### P1: DGX storage work root is configured but not prepared or systemd-whitelisted

Confirmed.

References:

- `deploy/dgx-spark/agentics.env.example:11`
- `crates/persistence/src/db/maintenance.rs:276`
- `deploy/dgx-spark/agentics-api.service:18`
- `ops/src/dgx_storage.rs:266`

Failure mode:

DGX env config sets `AGENTICS_STORAGE_WORK_ROOT=/srv/agentics/storage-work`. API startup challenge seeding and challenge publishing write tar scratch files there. The systemd unit uses `ProtectSystem=strict`, but only grants write access to `/srv/agentics/storage`. The DGX storage preparer also creates/chowns `storage`, `challenges`, and `runtime`, but not `storage-work`.

Impact:

Hosted API startup, seeded challenge sync, or challenge publishing can fail with permission or missing-directory errors on DGX.

Recommended fix:

Prepare and chown `/srv/agentics/storage-work`, add it to API and worker `ReadWritePaths`, and extend DGX profile checks to verify every configured writable path is prepared and whitelisted.

### P2: Local bundle tar creation is unbounded before storage limits apply

Confirmed.

References:

- `frontends/agentics-cli/src/commands.rs:804`
- `frontends/agentics-cli/src/commands.rs:815`
- `crates/storage/src/lib.rs:979`
- `crates/storage/src/lib.rs:1006`

Failure mode:

`pack_directory_to_tar` recursively writes a tar archive before `storage.put_file(..., challenge bundle archive)` enforces the configured byte cap. A large or hostile local bundle can fill the storage work disk before rejection.

Recommended fix:

Pass an archive/write policy into tar packing and enforce byte count, file count, and depth while appending through a counting writer. Abort and clean up before exceeding the cap.

### P2: S3 prefix deletion overmatches sibling prefixes

Confirmed.

References:

- `crates/storage/src/lib.rs:786`
- `crates/storage/src/lib.rs:789`
- `crates/storage/src/lib.rs:795`
- `crates/storage/src/lib.rs:817`

Failure mode:

S3 `list_prefix("_tmp")` lists physical prefix `_tmp`, so it also returns `_tmpfoo/...`. The default `delete_prefix` then deletes every returned key. Local storage behaves like a directory boundary and does not overmatch this way.

Impact:

Production code does not currently use `delete_prefix`, but it is the natural primitive for the stale `_tmp` cleanup that the docs now require. Using it as-is would be dangerous.

Recommended fix:

List directory-like prefixes with a trailing slash or post-filter logical keys to `key == prefix || key.starts_with(prefix + "/")`. Add RustFS tests proving `_tmpfoo` survives `delete_prefix("_tmp")`.

### P2: Docker wait-stream errors are reported as container exit code `1`

Confirmed.

References:

- `crates/runner/src/docker.rs:766`
- `crates/runner/src/docker.rs:778`

Failure mode:

`wait_container` results are collected and flattened. Any daemon or stream error is dropped, and the function falls back to exit code `1`.

Impact:

Infrastructure failures can be persisted as participant/evaluator process failures, producing misleading diagnostics and incorrect lifecycle classification.

Recommended fix:

Iterate wait events explicitly and return a Docker/platform error for any stream error. Use an exit-code fallback only when the stream completes successfully without a status.
