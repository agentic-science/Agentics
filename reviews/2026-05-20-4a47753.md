### Current Findings

#### P1: DGX worker bind mounts can point at the wrong `/tmp`

Diagnosis: hosted runner workspaces are created under `std::env::temp_dir()` and
then passed to Docker as bind-mount sources. The DGX worker service runs with
`PrivateTmp=true`, while Docker resolves bind-mount source paths in the Docker
daemon namespace. That can make the container fail to see the worker-created
path, or bind an empty path from the daemon namespace instead.

References:

- `backend/shared/src/runner.rs:295`
- `backend/shared/src/runner.rs:609`
- `backend/shared/src/runner.rs:707`
- `backend/shared/src/runner.rs:782`
- `deploy/dgx-spark/agentics-worker.service:17`
- `deploy/dgx-spark/agentics-docker.service:9`

Recommended fix: treat the DGX hosted runner path as Linux-only. Stop using
private temp paths for Docker bind mounts on Linux. Add an operator-owned
runtime root such as `/srv/agentics/runtime`, include it in the DGX worker
`ReadWritePaths`, and make the host probe create a file from the worker process
and verify the Agentics Docker daemon can bind-mount and read it. On macOS,
configuration that selects the DGX hosted profile or Linux-only quota-backed
runner storage should emit a clear configuration error instead of attempting to
start.

#### P2: Retained build and scorer input trees escape quota-backed storage

Diagnosis: setup/build and run phases execute in bounded writable slots, but the
worker then copies build workspaces and scorer-visible run trees back into
ordinary working-root directories. Those retained trees are later mounted into
dependent phases and are no longer directly governed by the XFS project quota
that bounded the original writable slot. This weakens the documented hosted
storage guarantee.

References:

- `backend/shared/src/runner.rs:299`
- `backend/shared/src/runner.rs:301`
- `backend/shared/src/runner.rs:629`
- `backend/shared/src/runner.rs:630`
- `backend/shared/src/runner.rs:733`
- `backend/shared/src/runner.rs:783`
- `backend/shared/src/runner/filesystem.rs:249`
- `docs/operations/en.md:110`

Recommended fix: keep the relevant quota-slot leases alive through dependent
run/scorer phases, or copy retained build and scorer input trees into
quota-managed runtime storage with equivalent byte and inode limits.

#### P2: CUDA targets expose all GPUs instead of the declared profile count

Diagnosis: CUDA target validation requires `resource_profile.hardware_metadata`
to include `gpu_count`, but the Docker runner ignores that value and always uses
Docker `DeviceRequest.count = -1`, which exposes all GPUs to the container. On a
host with more GPUs than a target declares, a submission can bypass GPU resource
accounting.

References:

- `backend/shared/src/challenge_bundle/images.rs:193`
- `backend/shared/src/models/challenge.rs:424`
- `backend/shared/src/runner/docker.rs:724`

Recommended fix: carry the validated hardware profile or effective GPU count
into `ContainerRequest` and set Docker `DeviceRequest.count` to that count.
Alternatively reject CUDA profiles whose declared GPU count cannot be enforced on
the host.

#### P2: Creator draft detail leaks admin checkout paths

Diagnosis: creator-owned draft detail routes return the raw `ChallengeDraftResponse`.
That DTO includes `validation_repository_path` and each validation record's
`repository_path`, both populated from admin/reviewer checkout paths. This
exposes server-local filesystem paths to challenge creators.

References:

- `backend/api-server/src/router.rs:101`
- `backend/api-server/src/challenge_creation_handlers.rs:109`
- `backend/api-server/src/challenge_creation_handlers.rs:377`
- `backend/shared/src/models/challenge_creation.rs:314`
- `backend/shared/src/models/challenge_creation.rs:345`

Recommended fix: split admin and creator draft projections. Creator-facing
responses should not reuse the raw admin `ChallengeDraftResponse` shape and
should omit `validation_repository_path` plus validation record
`repository_path`. Add integration coverage that creator-visible draft detail
after validation omits both fields.

#### P2: Rejection or abandonment can race active draft validation

Diagnosis: validation sets `active_validation_record_id` while a draft validation
is running. Reject and abandon transitions can move the draft to a terminal state
without checking or clearing that active validation pointer. When the validation
finishes afterward, it updates the validation record but can silently fail to
update the draft because the draft is no longer in a validation-compatible state.

References:

- `backend/shared/src/db/challenge_creation.rs:370`
- `backend/shared/src/db/challenge_creation.rs:540`
- `backend/shared/src/db/challenge_creation.rs:565`
- `backend/shared/src/db/challenge_creation.rs:611`
- `backend/shared/src/db/challenge_creation.rs:639`

Recommended fix: make terminal review transitions reject while
`active_validation_record_id IS NOT NULL`, or atomically clear the claim and mark
the active validation failed/cancelled. Treat a failed draft update in
`finish_challenge_draft_validation` as an explicit guarded state-machine outcome
instead of silent success.
