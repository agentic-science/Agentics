### P1: Run names accept `.` and `..` even though the runner uses them as path and mount components

Diagnosis: `RunName` uses the generic name-token syntax, which accepts any
non-empty token made from alphanumeric characters, `_`, `-`, or `.`. That means
`.` and `..` are valid run names. The runner joins the run name into
`runs_root`, deletes that path before materialization, and later builds a Docker
mount target from the same raw value. A challenge run manifest with `run_name:
".."` can make cleanup target the parent of the scorer-visible run directory and
can produce a container mount target such as `/solution-runs/..`.

References:

- `backend/shared/src/runner.rs:865`
- `backend/shared/src/runner/filesystem.rs:90`

Recommended fix: make `RunName` a safe path-component contract, or add a
runner-specific path-component newtype. Reject `.` and `..` explicitly and add
static and prepare-generated run manifest regression tests.

Status: resolved. `RunName` now rejects `.` and `..`, shared schemas reflect the
path-component rule, and run-manifest regression coverage was added.

### P1: DGX loopback hosted profiles can silently disable Docker writable-layer quotas

Diagnosis: `validate_runner_storage` requires
`AGENTICS_RUNNER_DOCKER_LAYER_QUOTA=true` for `xfs-project-quota-slots` only when
`AGENTICS_API_HOST` is non-loopback. The DGX environment binds the API to
`127.0.0.1` behind a public base URL. If the layer-quota flag is accidentally
removed or set false, hosted DGX workers can still pass config validation with
quota-backed bind mounts but unbounded writable Docker layers. The main runner
container keeps `readonly_rootfs=false`, so a participant can write outside the
bounded `/io` tree into the container writable layer.

### P2: Returning GitHub creators can be locked out after their one-use pioneer code is consumed

Diagnosis: GitHub OAuth start requires a pioneer code before redirecting to
GitHub and calls `ensure_pioneer_code_available`. That check rejects exhausted
codes. The callback-side DB path already allows an existing GitHub identity to
log in without consuming a code, but the start path rejects the exhausted code
before the GitHub identity is known.

Failure mode: a creator registered with a one-use code cannot start OAuth again
after their browser session expires, because the consumed code blocks the login
before the platform can determine that the GitHub user already exists.

References:

- `backend/api-server/src/auth_handlers.rs:173`
- `backend/api-server/src/auth_handlers.rs:186`
- `backend/shared/src/db/pioneer_codes.rs:220`
- `backend/shared/src/db/sessions.rs:133`
- `backend/shared/src/db/sessions.rs:185`

Recommended fix: make the login-start code optional for existing users. Store an
optional code hash in OAuth state, then in the callback first lock and look up
the GitHub identity. Existing active users should receive a session without
checking or consuming a code; new users should still require and consume a valid
code transactionally. Add an HTTP regression test for repeat creator login after
code exhaustion.

Status: resolved. GitHub OAuth start accepts an optional pioneer code, returning
GitHub creators can log in without re-entering a consumed code, and new creators
still require a valid code when pioneer-code registration mode is active.

### P2: Admin draft approval can approve a different validation digest than the reviewer saw, and audit is not atomic

Diagnosis: approval first reads the draft in the handler, then later approves
whatever `validation_bundle_sha256` is current in the database. The audit event
uses the digest from the stale pre-read. If another validation finishes between
the read and update, the approved digest can differ from the audit metadata.
Approval, rejection, abandonment, and validation completion also perform the
state transition and audit append as separate committed operations.

Failure mode: the API can return an error after the durable state transition has
already committed if audit insert fails. Retrying may conflict with the new
state, and audit history can be missing or point at the wrong validation digest.

### P2: Local storage symlink checks are TOCTOU guards, not enforced writes

Diagnosis: `LocalStorage::put` checks storage-key prefixes and final-object
symlinks, creates parent directories, checks again, and then calls
`tokio::fs::write`. `promote` repeats a similar check-then-link pattern before
`hard_link`. These checks reduce accidental symlink traversal, but they do not
bind the write or link to a stable directory handle.

Failure mode: if the storage root or one of its parents is writable by another
process, a checked path can be swapped for a symlink between validation and
write/link, causing storage writes or promotions outside the storage root.

References:

- `backend/shared/src/storage/mod.rs:187`
- `backend/shared/src/storage/mod.rs:189`
- `backend/shared/src/storage/mod.rs:193`
- `backend/shared/src/storage/mod.rs:195`

- `backend/shared/src/storage/mod.rs:207`
- `backend/shared/src/storage/mod.rs:213`
- `backend/shared/src/storage/mod.rs:216`

Recommended fix: replace check-then-open writes with root-relative safe file
operations using stable directory handles and no-follow semantics per component.
Use `create_new` for new object creation and atomic same-directory promotion.
Add regression coverage for final-object symlink and parent-swap behavior.

Status: mitigated. Local writes now use a same-directory temporary object opened
with `create_new`, reject final-object symlinks before writing, and atomically
rename into place. Final-object symlink regression coverage was added. Stable
directory-handle parent-swap hardening remains a post-MVP storage-root defense
if storage roots ever become writable by untrusted users.

### P2: Scorer `result.json` symlinks are followed in unbounded runner mode

Diagnosis: unbounded writable storage returns the fallback path directly. In
that mode the scorer output mount can be the same directory the worker later
reads. `replace_dir_all_if_separate` returns early when source and destination
match, so the symlink-rejecting copy path is skipped. `read_limited_result_json`
then uses `metadata` and `read_to_string`, both of which follow symlinks.

Failure mode: scorer code can create `/output/result.json` as a symlink and the
worker will follow it from the host namespace in unbounded local mode.

Status: resolved. `result.json` reads use `symlink_metadata` and reject symlinks

### P2: Bounded quota slots can be wedged by root-owned files after interrupted cleanup

Diagnosis: permission repair runs after container execution. Slot reuse removes
the previous `work` directory as the worker process, and lease-drop cleanup only
logs failures. If the worker dies after a root-running container creates
non-writable nested files but before permission repair, a future slot acquisition
can fail repeatedly when trying to remove that stale `work` tree.

References:

- `backend/shared/src/runner/docker.rs:141`
- `backend/shared/src/runner/storage.rs:140`
- `backend/shared/src/runner/storage.rs:251`
- `backend/shared/src/runner/storage.rs:253`
- `deploy/dgx-spark/agentics-worker.service:9`

Recommended fix: add startup and stale-slot repair using a privileged host helper
or hardened cleanup sidecar. Treat unrecoverable slot cleanup as
operator-visible capacity degradation instead of an ordinary evaluation failure.

Status: partially resolved. Unrecoverable slot cleanup is now capacity
degradation: the worker skips the slot, logs the cleanup failure, and requeues
the evaluation instead of failing participant work. A privileged host repair
helper remains a follow-up if DGX operations need automated repair beyond the
existing permission-repair sidecar.

bounded slot is busy. The worker turns runner errors into persisted failed
evaluations. Temporary slot exhaustion, or a stale live container that owns every
matching slot, can therefore fail otherwise valid submissions.

References:

- `backend/shared/src/runner/storage.rs:230`
- `backend/worker/src/cycle.rs:326`
- `backend/worker/src/cycle.rs:329`

Recommended fix: model slot exhaustion as retryable platform capacity. Requeue

submissions do not enter evaluation when no matching slot can be acquired.

Status: resolved. Bounded slot exhaustion is represented as runner capacity
pressure and running jobs are requeued with short backoff without consuming a
participant attempt.
