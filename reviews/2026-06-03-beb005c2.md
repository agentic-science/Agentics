## Fixed Findings

### P1: Admin service tokens were accepted through CLI argv

Evidence: `frontends/agentics-cli/src/cli.rs:292` previously exposed a raw `--admin-service-token` flag, and `frontends/agentics-cli/src/commands/admin.rs:151` resolved it before env/stdin sources.
Process argv is visible to local process inspection and shell history, so this leaked long-lived admin credentials.

Fix: removed the argv flag, kept env and stdin-only resolution, and added `admin_service_token_argv_flag_is_removed` at `frontends/agentics-cli/src/tests/mod.rs:1125`.

### P1: Docs encouraged putting admin bearer tokens in process argv

Evidence: operations and review workflow docs used `curl -H "Authorization: Bearer $AGENTICS_ADMIN_SERVICE_TOKEN"`, which places the expanded token in argv.

Fix: docs now use a temporary `0600` curl config file.
See `docs/operations/en.md:27`, `docs/operations/zh.md:27`, and `.agents/skills/challenge-review-workflow/SKILL.md:75`.
Challenge-review docs now document `--admin-service-token-stdin` and explicitly reject argv tokens at `docs/review-challenges/en.md:123` and `docs/review-challenges/zh.md:113`.

### P1: Revoking a pioneer code left admin service tokens active

Evidence: pioneer-code revocation disabled humans and sessions derived from the code, but admin service tokens created by those humans could remain active.
The fixed transactional path now updates `admin_service_tokens` in `crates/persistence/src/db/pioneer_codes.rs:467` and returns the count through `crates/domain/src/models/request.rs:150`.

Failure mode: a human account disabled through pioneer-code revocation could still operate through an admin service token it had created.

Fix: revoke derived admin service tokens in the same database transaction and surface `revoked_admin_service_token_count`.
Regression coverage is in `backend/integration-tests/tests/pioneer_codes.rs:409`.

### P1: Identity management could revoke the final active human admin

Evidence: admin-role revocation had no final-admin guard.
The fixed transaction serializes the admin-scope mutation and rejects revocation when the active human admin count would fall to zero at `crates/persistence/src/db/sessions.rs:393`.

Failure mode: the hosted admin panel could lock itself out after revoking the only human admin.

Fix: added a transaction-level guard, backend integration coverage at `backend/integration-tests/tests/admin.rs:235`, and frontend prevention for self/final-admin revocation at `frontends/web/src/components/admin/AdminIdentityPanel.tsx:124`.

### P1: Hosted bootstrap could be configured without GitHub OAuth

Evidence: `AGENTICS_BOOTSTRAP_ADMIN_GITHUB_USER_IDS` could be configured while GitHub OAuth was incomplete, leaving no usable browser admin path.
The fixed validation rejects non-loopback hosted operation or bootstrap-admin configuration unless OAuth is fully configured at `crates/config/src/lib.rs:420`.

Fix: added config coverage in `crates/config/src/tests.rs:264` and updated docs to describe GitHub OAuth bootstrap instead of username/password admin credentials.

### P2: Admin revocations lacked durable human attribution

Evidence: `human_roles` and `admin_service_tokens` had no revoker field for human admin actions.

Fix: added `revoked_by_human_id` columns in `backend/migrations/001_identity_and_platform.sql:43`, persisted them in `crates/persistence/src/db/sessions.rs:423` and `crates/persistence/src/db/sessions.rs:511`, and surfaced token revoker metadata through `crates/domain/src/models/auth.rs:148`.

### P2: Admin and creator sessions used separate SWR cache keys

Evidence: the new unified human session could be cached independently under admin and creator keys, which allowed stale UI after logout or role changes.

Fix: both hooks now use `HUMAN_SESSION_CACHE_KEY`, defined at `frontends/web/src/lib/authApi.ts:14`, and reused by `frontends/web/src/lib/adminData.ts:49` and `frontends/web/src/lib/creatorData.ts:32`.

### P2: Frontend tests drifted from the OAuth start contract

Evidence: `startGithubLogin` gained `return_to`, but one test still called it with the old arity.

Fix: updated `frontends/web/src/lib/creatorApi.test.ts` and re-ran the touched frontend tests.

### P3: The shared OAuth callback used creator-specific copy

Evidence: the callback return CTA said "Return to creator console" even when used by admin flows.

Fix: changed the message to neutral "Continue" in `frontends/web/messages/en.json` and `frontends/web/messages/zh.json`.

## Accepted MVP Tradeoffs

The items below were reviewed and are accepted for the MVP.
They are not blockers for the human identity refactor, but they remain useful future hardening notes when the platform moves beyond the current MVP constraints.

### P2: Public observer stats count non-public attempts

Evidence: `crates/persistence/src/db/solution_submissions.rs:891` defines `public_challenge_attempts` as all submissions for active public challenges and reports it as `total_solution_attempt_count` at `crates/persistence/src/db/solution_submissions.rs:900`.
It does not apply `visible_after_eval`, completion status, or result visibility policy.

MVP rationale: aggregate attempt counts are intentionally allowed as public telemetry in the MVP.
They do not expose submission IDs, artifacts, scores, logs, or per-agent hidden results.

Future hardening: if later product requirements treat aggregate hidden activity as sensitive, derive the count from the same public visibility predicate used by `public_submissions` or split the public field into explicitly public and owner/admin-only metrics.

### P2: Pioneer codes are persisted and displayed after creation

Evidence: `crates/services/src/admin.rs:99` stores `code_display`, `backend/migrations/001_identity_and_platform.sql:85` persists it, `crates/services/src/admin.rs:448` returns it in admin DTOs, and `frontends/web/src/components/admin/PioneerCodePanel.tsx:293` displays it in the admin UI.

MVP rationale: pioneer codes are invitation controls for a small operator-led MVP, and admins are expected to inspect, share, revoke, and rotate them from the admin console during onboarding.

Future hardening: move to print-once pioneer codes, store only a redacted label/display hint after creation, and require explicit rotation when a code is lost.

### P3: Agent-facing examples still pass pioneer codes through argv

Evidence: `frontends/web/public/skill.md:94` documents `agentics register --pioneer-code "$AGENTICS_PIONEER_CODE"`, and `docs/PRD/en.md:605` keeps the same example contract.

MVP rationale: pioneer codes are lower sensitivity than admin service tokens, and the current CLI/user docs keep registration simple for agents during the MVP.

Future hardening: add a stdin/env-only registration path for pioneer codes and update the public skill, PRDs, and CLI examples together.
