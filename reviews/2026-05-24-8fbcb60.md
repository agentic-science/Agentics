## Findings

### P1: Public response schemas strip unknown fields, masking redaction regressions

Refs:

- `frontends/web/src/lib/schemas.test.ts:482`
- `frontends/web/src/lib/generated/schemas.ts:2758`

Problem: the web schema redaction tests pass intentionally private fields such as `aggregate_metrics` into public DTO parsers and then assert that the parsed result omitted those keys.
The generated Zod schemas use the default object behavior, which strips unknown properties.
That means the tests can pass even if the backend starts returning those private fields on public endpoints.

Impact: a backend redaction regression on public challenge, submission, leaderboard, ranking, or report DTOs can be hidden by frontend parsing instead of failing loudly.

Fix: make public response schemas strict, or emit `additionalProperties: false` for DTOs where unknown keys should be contract violations.
Update the tests to assert that private metric payloads are rejected rather than silently stripped, while keeping the backend as the source of public projection/redaction.

### P2: Duplicate separated-evaluator run input paths overwrite each other

Refs:

- `backend/shared/src/challenge_bundle.rs:759`
- `backend/shared/src/runner/run_io.rs:237`

Problem: session manifests reject duplicate materialized input paths, but separated-evaluator run manifests do not.
During run input materialization, multiple inputs with the same `path` can be processed in order and later entries overwrite earlier entries.

Impact: a challenge author can accidentally publish ambiguous run inputs, and the final evaluator input depends on ordering instead of an explicit, validated contract.

Fix: add duplicate normalized-path validation for `RunInputSpec.path`, matching the session manifest policy.
Keep runner materialization defensive by rejecting pre-existing materialized destinations and using overwrite-resistant creation where applicable.

Decision: fix by adding a duplicate-path `HashSet` check for separated-evaluator run input files and by rejecting pre-existing materialized destination paths during runner input staging.

### P2: Creator OAuth state and CSRF tokens are stored in browser `sessionStorage`

Refs:

- `frontends/web/src/lib/creatorApi.ts:34`
- `frontends/web/src/lib/creatorApi.ts:73`
- `frontends/web/src/lib/creatorApi.ts:79`
- `frontends/web/src/lib/creatorApi.ts:86`
- `frontends/web/src/lib/creatorApi.ts:97`
- `frontends/web/src/components/creator/CreatorConsole.tsx:164`
- `frontends/web/src/components/creator/CreatorOAuthCallback.tsx:36`

Problem: creator OAuth state and the creator CSRF token are persisted in `sessionStorage`.
The backend already has OAuth state storage, and CSRF tokens should not be placed in browser storage when they can be held in component memory or an HttpOnly/SameSite flow.

Impact: an XSS or browser-storage leak can expose creator CSRF material and make OAuth state handling more fragile than necessary.

Fix: remove `sessionStorage` from the creator auth flow.
Let the callback send the `state` value from the Moltbook/GitHub redirect back to the backend for one-time validation, and keep CSRF only in React memory after `/api/creator/session` or callback completion.

### P2: CLI lossy-converts admin repository paths before API submission

Refs:

- `frontends/agentics-cli/src/commands.rs:135`
- `frontends/agentics-cli/src/commands.rs:192`

Problem: CLI challenge-draft creation and update convert repository paths with `to_string_lossy()`.
Non-UTF-8 bytes can be replaced with U+FFFD before they are sent to the API.

Impact: malformed local paths can become different wire values, producing confusing validation failures or pointing the backend at a path that was not exactly requested.

Fix: reject non-UTF-8 repository paths explicitly with a clear CLI error, or introduce a typed path boundary that preserves the exact accepted contract.
Do not use lossy conversion for API request fields.

Decision: reject non-UTF-8 paths explicitly with a clear error before constructing the admin challenge-draft API request.
