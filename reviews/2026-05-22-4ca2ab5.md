### P0: DGX purge path validation can delete broad system trees

Refs:

- `ops/src/dgx_profile.rs:178`
- `ops/src/dgx_profile.rs:233`
- `ops/src/dgx_profile.rs:453`
- `ops/src/dgx.rs:387`

Problem: `uninstall --purge-data` validates only `state_root` and `test_state_root`, and it treats each configured root as its own allowed root.
It never validates `config_root` or `release_root`, but the purge plan recursively removes `config_root`, `release_root`, `state_root`, and `test_state_root`.
With root, `AGENTICS_DGX_PROFILE_CONFIRM=uninstall-purge`, and an env override such as `AGENTICS_DGX_CONFIG_ROOT=/etc`, `AGENTICS_DGX_RELEASE_ROOT=/opt`, or `AGENTICS_DGX_STATE_ROOT=/srv`, the Rust binary can recursively delete broad system-owned trees.

Impact: a single misconfigured destructive command can remove host configuration, release artifacts, or unrelated services.

Fix: validate every purge root against fixed safe defaults or narrowly allowed children.
Reject top-level directories, symlinks, parent traversal, and `state_root == test_state_root`.
Add hostile env override tests for every root removed by purge.

### P1: Local demo can migrate and seed a non-demo database

Refs:

- `ops/src/local_demo.rs:339`
- `ops/src/local_demo.rs:633`
- `ops/src/local_demo.rs:654`
- `ops/src/local_demo.rs:670`
- `scripts/dev/local-demo.sh:154`

Problem: the Rust demo falls back to ambient `AGENTICS_DATABASE_URL` when `AGENTICS_DEMO_DATABASE_URL` is absent.
It then resets `agentics_demo`, but migrations and seed SQL run against `config.database_url`.
The shell script always overwrites the generic database URL with a demo URL, so ambient foreground-development config cannot accidentally receive demo migrations or fake data.
The Rust path also ignores `start_container` errors, so an already-running or external database at the ambient URL can be used without proving the demo container is the target.

Impact: `agentics-local-demo up` can run migrations and seed fake demo rows into a real local development database.

Fix: never consume generic `AGENTICS_DATABASE_URL` for the demo.
Use a generated local demo URL or a validated `AGENTICS_DEMO_DATABASE_URL`, require a loopback host unless explicitly confirmed, require the URL database path to match `AGENTICS_DEMO_DATABASE_NAME`, and fail if the demo Postgres container did not start or is not owned by the demo.

### P1: Local demo seed SQL is executed through the wrong SQLx API

Refs:

- `ops/src/local_demo.rs:676`
- `ops/src/local_demo_seed.sql:1`
- `ops/src/local_demo_seed.sql:11`
- `scripts/dev/local-demo.sh:368`

Problem: `sqlx::query(DEMO_SEED_SQL)` is used for a file containing many top-level SQL statements.
The shell sent the heredoc to `psql`, which supports scripts.
SQLx prepared queries do not reliably accept a multi-statement script as one prepared statement.

Impact: `agentics-local-demo up` can fail at seeding after starting services and resetting/migrating the database.

Fix: execute the seed with `sqlx::raw_sql(DEMO_SEED_SQL).execute(&pool).await?` or split the file into explicit statements.
Add a migrated-DB seed smoke test.

### P1: Local demo persists env-supplied admin passwords and ignores saved generated passwords

Refs:

- `ops/src/local_demo.rs:379`
- `ops/src/local_demo.rs:419`
- `ops/src/local_demo.rs:835`
- `scripts/dev/local-demo.sh:79`

Problem: the Rust config stores any non-default `AGENTICS_ADMIN_PASSWORD` as the demo password, and `ensure_admin_password` writes that value to `.agentics-demo/admin-password` when the file is absent.
The shell only persisted generated demo passwords, read the existing generated file on later runs, and attempted `chmod 600`.
The Rust path also does not read an existing generated password file back into `config.admin_password`, so repeated runs can rotate the in-process password while the saved file says something else.

Impact: operator-provided secrets can be persisted to disk unexpectedly, and the demo password file can become misleading.

Fix: track password source explicitly. Never persist env-supplied secrets. If a generated password file exists, read it and use it.
Create generated password files with `0600`, `create_new`, and symlink rejection.

### P1: Local demo `down` can leave API or web child processes alive

Refs:

- `ops/src/local_demo.rs:750`
- `ops/src/local_demo.rs:771`
- `ops/src/local_demo.rs:889`
- `scripts/dev/local-demo.sh:256`
- `scripts/dev/local-demo.sh:275`

Problem: the Rust demo spawns `cargo` and `bun` directly, stores only the wrapper PID, and `terminate_pid` signals only that PID.
The shell started each service in a new session and killed the process group.
`cargo run` and `bun run dev` can leave child API or Next.js processes behind after the wrapper exits.

Impact: `agentics-local-demo down` can report success while ports remain occupied by orphaned children.

### P1: DGX install copies secret env files without explicit mode or ownership

Refs:

- `ops/src/dgx_profile.rs:383`
- `ops/src/dgx_profile.rs:395`
- `scripts/ops/manage-dgx-spark-profile.sh:86`
- `deploy/dgx-spark/agentics.env.example:7`

Problem: Rust creates the config directory then copies `agentics.env` without setting `root:<service_group>` ownership or `0640` mode.
The shell installed `/etc/agentics` as `root:$SERVICE_GROUP` and `agentics.env` as `0640 root:$SERVICE_GROUP`.
The copied template becomes the real env file that contains database, admin, and OAuth secret fields.

Impact: file permissions depend on source checkout mode and process umask.
A host can end up with a secret env file that is too broadly readable.

Fix: use explicit Unix ownership and mode for copied files, especially `0640 root:<service_group>` for `agentics.env`, and repair permissions even when the file already exists.

### P1: Destructive path checks are lexical and allow `..` escapes

### P2: DGX storage can clobber `/etc/fstab` on read errors

Refs:

- `ops/src/dgx_storage.rs:447`
- `ops/src/dgx_storage.rs:456`
- `ops/src/dgx_storage.rs:590`

Problem: storage preparation treats any `/etc/fstab` read error as an empty file, appends the new line to that empty string, and writes it back.
Rollback has the same empty fallback. The shell only appended to the file.
This is especially risky because `/etc/fstab` is not guaranteed to be UTF-8.

Impact: an I/O or UTF-8 error can overwrite existing fstab content and break host boot or mount behavior.

Fix: read and back up bytes, propagate errors except `NotFound`, append atomically from a verified backup, and store rollback backups as bytes.

### P2: DGX profile fstab cleanup removes lines by broad substring

### P2: Bounded output is still buffered unboundedly

Refs:

- `ops/src/support.rs:244`
- `ops/src/support.rs:301`
- `ops/src/check_dgx_spark_host.rs:354`
- `ops/src/check_dgx_spark_profile.rs:578`
- `backend/worker/src/host_probe.rs:35`

Problem: `run_process` reads stdout and stderr to EOF before applying `limit_bytes`, Docker log collectors append all log chunks, and the worker host probe uses `Command::output()` with no cap.
The display cap happens after memory has already been allocated.

Impact: a noisy host tool, bad probe binary, or overridden Docker image can exhaust memory before truncation.

Fix: stream stdout/stderr/logs with a hard byte cap, mark truncation, keep draining or kill safely after the cap, and wrap worker probe execution in a startup timeout.

### P2: Ctrl-C can leave rootful child processes or probe containers behind

Refs:

- `ops/src/support.rs:167`
- `ops/src/support.rs:231`
- `ops/src/dgx_storage.rs:65`
- `ops/src/dgx_profile.rs:91`
- `ops/src/check_dgx_spark_host.rs:327`
- `ops/src/check_dgx_spark_profile.rs:552`

Problem: `run_with_ctrl_c` returns by dropping the active future, but spawned commands are not configured with `kill_on_drop(true)` or process-group cleanup.
Docker probe timeout paths also return before force-removing the created container.

Impact: interrupted rootful storage/profile commands can keep mutating the host after the CLI exits, and timed-out probes can leak containers.

Fix: use kill-on-drop or process groups for external commands, wire cancellation through rollback paths, and always force-remove created Docker containers in timeout/error cleanup.
