## P1 Findings

### P1: S3-default integration tests still assert durable objects on local temp files

Confirmed.

References:

- `backend/integration-tests/tests/helpers.rs:125`
- `backend/integration-tests/tests/challenge_creation_assets.rs:70`
- `backend/integration-tests/tests/challenge_creation/lifecycle.rs:596`

Failure mode:

`test_config` now defaults integration tests to `StorageBackend::S3`, but some tests still validate durable upload success by checking `storage.path().join(storage_key).exists()`. With S3/RustFS, successful uploads write to object storage, not to the local tempdir returned by the test helper.

Impact:

These tests will fail under the intended RustFS-backed integration environment, or they will push maintainers toward local-backend workarounds that undermine the new default.

Recommended fix:

Keep the test `Config` available where durable-storage assertions are needed and replace local-path checks with storage abstraction checks such as `build_storage(&config).await?.exists(...)`, `get(...)`, `get_to_file(...)`, or `list_prefix(...)`.

## P2 Findings

### P2: S3 promotion materializes private objects with default filesystem permissions

Confirmed.

References:

- `crates/storage/src/lib.rs:679`
- `crates/storage/src/lib.rs:784`

Failure mode:

`S3Storage::promote` downloads promoted objects into `work_root`. The work directory and temporary file are created with default OS permissions. On common Unix umasks, private challenge assets, solution ZIPs, or runner artifacts can be briefly materialized as group-readable or world-readable local files.

Impact:

Moving the durable default to S3 still leaves a local scratch exposure path for private data during promotion.

Recommended fix:

Create `AGENTICS_STORAGE_WORK_ROOT` with restrictive permissions, use `0600` temp files on Unix, reject or warn on unsafe existing work-root permissions, and add a regression test for temp-file modes.

### P2: S3 promotion can report failure after committing the durable object

Confirmed.

References:

- `crates/storage/src/lib.rs:684`
- `crates/storage/src/lib.rs:687`
- `crates/storage/src/lib.rs:691`
- `crates/storage/src/lib.rs:696`

Failure mode:

If `get_to_file`, `put_file`, and deletion of the temporary S3 object succeed, but local scratch cleanup fails, `promote` returns an error after the durable object has already been written and the temporary object may already be gone.

Impact:

Callers can treat a committed promotion as failed, which can lead to DB cleanup, orphaned durable objects, or non-idempotent retry behavior.

Recommended fix:

Do not report a committed remote promotion as failed solely because scratch cleanup failed. Either reorder cleanup so it cannot invalidate a committed promote result, or downgrade scratch cleanup failure to a warning-style cleanup error that does not hide the committed durable key. Add a test with injected cleanup failure.

### P2: Stale durable-object integration coverage no longer exercises S3

Confirmed.

References:

- `backend/integration-tests/tests/challenge_creation_assets.rs:379`
- `backend/integration-tests/tests/challenge_creation_assets.rs:382`
- `backend/integration-tests/tests/challenge_creation_assets.rs:426`

Failure mode:

The stale private-asset cleanup test writes a stale durable object directly into the local tempdir and later reads that local file. Under S3/RustFS, this no longer seeds or verifies the configured durable-storage backend.

Impact:

The test can pass while the S3 cleanup path is broken or leaking stale private asset objects.

Recommended fix:

Seed the stale object through `Storage::put` or another configured-storage helper before inserting and aging the DB row, then verify the replacement through `Storage::get` or `exists`.

### P2: Cleanup tests still inspect local paths for S3 durable-storage absence

Confirmed.

References:

- `backend/integration-tests/tests/request_validation.rs:309`
- `backend/integration-tests/tests/challenge_creation.rs:980`
- `backend/integration-tests/tests/challenge_creation.rs:990`

Failure mode:

Several cleanup tests assert absence by inspecting local directories under `storage.path()`. With S3/RustFS as the default backend, those local paths are not the durable object store.

Impact:

The tests can pass while S3 objects are leaked after invalid submissions or failed challenge publish flows.

Recommended fix:

Use `Storage::list_prefix`, `exists`, or typed storage-key checks for durable object absence. Keep `storage_work_root(config)` checks only for local staging and scratch-directory cleanup.
