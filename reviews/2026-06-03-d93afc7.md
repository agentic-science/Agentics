## Active Findings

### P2: HTTP GitHub sign-in examples set `Secure` cookies and can break callback state

Evidence:

- `deploy/compose/env/dev.env.example:41` sets `AGENTICS_WEB_BASE_URL=http://localhost:3001`, while `deploy/compose/env/dev.env.example:50` sets `AGENTICS_WEB_SESSION_COOKIE_SECURE=true`.
- `deploy/compose/env/rehearsal.env.example:59` also sets `AGENTICS_WEB_SESSION_COOKIE_SECURE=true`, while `deploy/compose/env/rehearsal.env.example:64` uses `http://127.0.0.1:13001/creator/github/callback`.
- The callback handler requires the browser nonce cookie at `backend/api-server/src/auth_handlers.rs:86`.
- Cookie formatting appends `Secure` whenever the configured flag is true at `backend/api-server/src/auth_handlers.rs:213`.

Failure mode:

Local dev and disposable rehearsal GitHub sign-in can redirect back over plain HTTP without the nonce cookie. That causes `/api/auth/github/callback` to reject the callback as unauthorized. The issued session cookies have the same persistence problem.

Remediation:

Either set `AGENTICS_WEB_SESSION_COOKIE_SECURE=false` in HTTP loopback dev and rehearsal examples, or require HTTPS for those examples. Add config validation or a focused test so `web_session_cookie_secure=true` with an HTTP loopback callback is not accidentally presented as a working default.

### P2: GitHub App URL validation accepts unsafe redirect and endpoint shapes

Evidence:

- `crates/domain/src/models/urls.rs:633` allows GitHub App redirect URLs with either `http` or `https`.
- `crates/config/src/lib.rs:397` validates that GitHub App fields are present together, but does not enforce HTTPS redirects outside loopback development.
- `crates/domain/src/models/urls.rs:761` validates exact HTTPS GitHub authorize/token URLs by scheme, host, port, and path, but unlike the GitHub base validator at `crates/domain/src/models/urls.rs:776`, it does not reject URL userinfo.
- The authorize URL is returned to browsers at `crates/services/src/auth.rs:385`, and the token URL is used for the code exchange at `crates/services/src/auth.rs:115`.

Failure mode:

A hosted or reverse-proxied deployment can pass startup validation with a non-loopback `http://...` GitHub redirect URL, risking authorization-code exposure or runtime login failure. URLs such as `https://user:pass@github.com/login/oauth/authorize` also satisfy the current exact-host checks and would put credential-bearing URL text into browser redirects or outbound requests.

Remediation:

Reject username/password userinfo for all GitHub App URL wrappers. Require HTTPS redirect URLs except explicit loopback callbacks for local development. Add URL wrapper tests and config policy tests for non-loopback HTTP redirects and userinfo-bearing authorize/token URLs.

### P3: Challenge catalog query validation is split between HTTP handlers and persistence

Evidence:

- `backend/api-server/src/handlers.rs:198` defines handler-local `ChallengeCatalogQuery`.
- `backend/api-server/src/handlers.rs:242` performs search and keyword validation before building `ChallengeCatalogFilters`.
- Persistence owns the lower-level catalog filter model in `crates/persistence/src/db/challenges/records.rs`.
- Shared public API validation currently covers pagination in `crates/contracts/src/validation/public_api.rs`, but not the full catalog query.

Failure mode:

Future API, CLI, or service call paths can bypass handler-only search length and keyword-count limits before persistence executes expensive text filters.

Remediation:

Introduce a typed `PublicChallengeCatalogQuery::try_new` in `agentics-contracts` or `agentics-domain`. Move query parsing, search normalization, keyword parsing, keyword count, limit, and offset validation into that type. Services should accept the typed value, and persistence should consume already-validated SQL parameters.

### P3: `CreatorConsole` keeps duplicate owners for remote data

Evidence:

- `frontends/web/src/components/creator/CreatorConsole.tsx:66` through `:106` stores creator session, review record, stats, participants, shortlist, and owner scope in component state.
- The same remote resources are also loaded through SWR hooks at `frontends/web/src/components/creator/CreatorConsole.tsx:118` through `:120`.
- Mutations manually fetch and set records at `frontends/web/src/components/creator/CreatorConsole.tsx:319` through `:324` and `:347` through `:350`, then also mutate SWR caches.

Failure mode:

Displayed creator data can diverge from the SWR cache after failed refreshes, concurrent updates, or mutation responses. Tests also need to mock both direct API calls and cache behavior.

Remediation:

Let `frontends/web/src/lib/creatorData.ts` own remote creator session, review record, owner bundle, refresh, and mutation cache updates. Keep `CreatorConsole` state limited to forms, pending action, and transient messages.

### P3: Admin CSRF tokens are used as SWR cache keys

Evidence:

- `frontends/web/src/components/admin/AdminConsole.tsx:41` stores the CSRF token in React state.
- `frontends/web/src/components/admin/AdminConsole.tsx:49` passes it into `useAdminDashboard`.
- `frontends/web/src/lib/adminData.ts:162` embeds the token in the SWR key.

Failure mode:

An authorization token becomes cache identity metadata, increasing exposure through browser devtools, debugging hooks, serialized SWR diagnostics, or future snapshots.

Remediation:

Use a non-secret cache key such as `["admin-dashboard"]` plus a non-secret session version or human ID if needed. Pass the CSRF token only through the fetcher closure or request options.

### P3: GitHub user IDs remain raw `i64` across auth and authorization boundaries

Evidence:

- `crates/services/src/auth.rs:50` stores `GithubSignInUser.id` as `i64`.
- `crates/persistence/src/db/sessions.rs:63` stores `ResolveGithubHumanInput.github_user_id` as `i64`.
- `crates/config/src/groups.rs:97` stores bootstrap admin GitHub user IDs as `Vec<i64>`.
- `backend/migrations/001_identity_and_platform.sql:35` stores `provider_user_id BIGINT NOT NULL` without a positive-ID check.

Failure mode:

The compiler cannot distinguish GitHub user IDs from arbitrary integers across bootstrap, session, PR-author, and persistence flows. Positivity is currently validated only on some external paths.

Remediation:

Introduce a `GithubUserId` domain newtype with positive validation. Use it in config, auth DTOs, service inputs, persistence records, and DB bind/read helpers. Add a DB `CHECK (provider_user_id > 0)`.

### P3: Real GitHub callback route is not integration-testable without live GitHub

Evidence:

- The HTTP handler hardcodes `ReqwestGithubSignInClient` at `backend/api-server/src/auth_handlers.rs:89`.
- Cookie issuance, nonce clearing, callback body handling, and session issuance happen in the same handler at `backend/api-server/src/auth_handlers.rs:98`.
- Existing integration coverage checks login start and lower-level state consumption, but not a successful `/api/auth/github/callback` route with a fake GitHub user.

Failure mode:

Cookie issuance, nonce expiry, callback JSON contract, disabled-human handling, and session creation can regress without a route-level integration test catching it.

Remediation:

Inject an `Arc<dyn GithubSignInClient>` or a small auth-service facade through app state. Use a fake client in integration tests for successful callback, invalid nonce, disabled human, pioneer-code-required, and state-reuse paths.

## Accepted MVP Risks Re-Observed

### Pioneer-code secrets are durably stored and re-displayed

Evidence:

- Generated codes use 4 random bytes at `crates/domain/src/models/pioneer_codes.rs:146`.
- Raw display values are stored in `code_display` at `crates/services/src/admin.rs:99` and persisted by `backend/migrations/001_identity_and_platform.sql:87`.

This matches the previously accepted MVP risk around plaintext and re-exposed pioneer codes. The secure direction remains one-time display, stronger entropy, and persisted hashes or fingerprints only.

### Permission-repair containers trust workload image userspace

Evidence:

- The runner captures `request.image` for permission repair at `crates/runner/src/docker.rs:116`.
- It runs `sh -c chmod -R ...` from that image at `crates/runner/src/docker.rs:264`.
- The helper adds `CAP_FOWNER` at `crates/runner/src/docker.rs:635`.

Hosted digest/repository image policy mitigates this, but a compromised approved/local workload image can still extend trust into platform cleanup. The stronger post-MVP shape is a platform-owned pinned repair image or host-side ownership repair.

### Public stats intentionally include broader challenge activity

Evidence:

- `crates/persistence/src/db/solution_submissions.rs:877` defines `public_submissions` with `visible_after_eval`, completed status, and result-detail visibility predicates.
- `crates/persistence/src/db/solution_submissions.rs:891` defines `public_challenge_attempts` as every submission for active public challenges, without the same visibility predicate.
- `backend/integration-tests/tests/public_read.rs:239` proves a pending submission is 404 publicly, while `backend/integration-tests/tests/public_read.rs:260` asserts that the same pending attempt is included in public `total_solution_attempt_count`.

Decision:

This is expected for MVP. Public aggregate stats may intentionally show broader platform activity than public submission detail/list surfaces. The current SQL-owned implementation is accepted for now.
